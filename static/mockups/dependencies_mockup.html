Связи между задачами (Dependencies)

Анализ вариантов реализации для Системы управления планированием

Типы связей (как в MS Project)

4 типа связей между задачами
FS — Финиш-Старт
Задача B начинается после окончания задачи A
Самый распространённый тип (~90% связей)
SS — Старт-Старт
Задача B начинается одновременно с задачей A
Параллельная работа с общим стартом
FF — Финиш-Финиш
Задача B заканчивается вместе с задачей A
Синхронное завершение работ
SF — Старт-Финиш
Задача B заканчивается при старте задачи A
Редкий тип (сменная работа, замена)
Lag (задержка) — положительное смещение: B начинается через N дней ПОСЛЕ связи.
Lead (опережение) — отрицательное смещение: B начинается за N дней ДО окончания связи.
Пример: FS + lag 3 дня = «B начинается через 3 дня после окончания A»

Рекомендация: комбинированный подход (A + B)

РЕКОМЕНДАЦИЯ Чипы в таблице + drawer по клику + валидация

Совмещаем компактность варианта A с информативностью варианта B:

  • В таблице: узкая колонка с компактными чипами (только «FS #1», «SS #3»)
  • По клику на чип: боковая панель с полными деталями
  • По клику +: модальное окно добавления связи с поиском
  • Автоматическая валидация: предупреждение о циклах и нарушении дат

Архитектура (бэкенд)

Модель данных

class TaskDependency(models.Model):
    """Связь между двумя задачами Work (predecessor → successor)."""

    # Типы связей (как в MS Project)
    LINK_TYPES = [
        ('FS', 'Финиш → Старт'),
        ('SS', 'Старт → Старт'),
        ('FF', 'Финиш → Финиш'),
        ('SF', 'Старт → Финиш'),
    ]

    predecessor = models.ForeignKey(
        Work, on_delete=models.CASCADE,
        related_name='successor_links',  # work.successor_links.all() → кто зависит от меня
        verbose_name='Предшественник',
    )
    successor = models.ForeignKey(
        Work, on_delete=models.CASCADE,
        related_name='predecessor_links',  # work.predecessor_links.all() → от кого я завишу
        verbose_name='Последователь',
    )
    link_type = models.CharField(
        'Тип связи',
        max_length=2,
        choices=LINK_TYPES,
        default='FS',
    )
    lag_days = models.IntegerField(
        'Задержка (дни)',
        default=0,
        help_text='Положительное = задержка, отрицательное = опережение',
    )

    class Meta:
        db_table = 'work_task_dependency'
        unique_together = [('predecessor', 'successor')]
        verbose_name = 'Связь задач'

    def clean(self):
        # Запрет связи на самого себя
        if self.predecessor_id == self.successor_id:
            raise ValidationError('Задача не может зависеть от самой себя')

Валидация на сервере

def detect_cycle(successor_id, predecessor_id):
    """BFS/DFS: проверяем, что добавление связи не создаёт цикл."""
    visited = set()
    queue = [predecessor_id]
    while queue:
        current = queue.pop(0)
        if current == successor_id:
            return True  # Цикл найден!
        if current in visited:
            continue
        visited.add(current)
        # Поднимаемся по цепочке предшественников
        preds = TaskDependency.objects.filter(
            successor_id=current
        ).values_list('predecessor_id', flat=True)
        queue.extend(preds)
    return False

План поэтапной реализации

Этап 1 — Фундамент (бэкенд + миграция)

  • Модель TaskDependency + миграция
  • API: CRUD для связей
  • Валидация: запрет циклов (BFS), запрет самоссылки, unique_together
  • Включение связей в сериализацию _serialize_task()

Этап 2 — UI: колонка + чипы (вариант A)

  • Колонка «Предш.» в таблице СП
  • Чипы с цветовой маркировкой типа связи
  • Кнопка «+» для добавления → модальное окно
  • Удаление по крестику на чипе

Этап 3 — Drawer + валидация дат (вариант B)

  • Боковая панель по клику на чип
  • Предупреждения при нарушении дат (FS: successor.start < predecessor.end)
  • Кнопка «Скорректировать дату» — автоматическое выравнивание

Этап 4 (будущее) — Ганта (вариант C)

  • Отдельная вкладка/режим с диаграммой Ганта
  • SVG-стрелки между барами
  • Критический путь

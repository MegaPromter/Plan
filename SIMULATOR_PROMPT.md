# Промт для создания программы-симулятора работы НТЦ

## Задача

Создать автономную Python-программу, которая симулирует работу подразделений научно-технического центра (НТЦ) в течение 1 года. Симуляция ускорена: 1 год виртуального времени = максимум 1 час реального времени. Программа тестирует Django-приложение planapp_django через его REST API.

## Контекст системы

Приложение planapp_django — система планирования работ НТЦ:
- **Организационная структура**: НТЦ-центры → Отделы → Секторы → Сотрудники
- **Иерархия данных**: Проект (УП) → Производственный план (ПП) → Задачи (СП) → Отчёты → Извещения
- **Роли**: admin, ntc_head, ntc_deputy, dept_head, dept_deputy, sector_head, user
- **Типы работ**: "Выпуск нового документа", "Корректировка документа", "Техническое сопровождение" и др.

## API Endpoints (базовый URL: http://localhost:8001)

### Аутентификация
- `POST /api/login/` — вход (username, password) → получение сессии
- `POST /api/register_public/` — регистрация (username, password, last_name, first_name, patronymic, department, sector, position)

### Проекты УП
- `GET/POST /api/projects/` — CRUD проектов (name_full, name_short, code)
- `GET/PUT/DELETE /api/projects/<id>/` — деталь проекта
- `GET/POST /api/projects/<id>/products/` — изделия проекта (name, code)

### Производственные планы (ПП)
- `GET/POST /api/pp_projects/` — CRUD ПП-планов (name, up_project_id)
- `GET /api/production_plan/?project_id=<pp_id>&limit=500` — список строк ПП
- `POST /api/production_plan/` — создание строки ПП (JSON: work_name, work_designation, date_start, date_end, department, sector, executor, task_type, row_code, work_order, stage_num, milestone_num, work_num, sheets_a4, norm, coeff, total_2d, total_3d, labor, pp_project)
- `PUT /api/production_plan/<id>/?field=<fieldname>` — inline-обновление поля (body: {value: ...})
- `DELETE /api/production_plan/<id>/` — удаление строки

### Синхронизация ПП→СП
- `POST /api/tasks/sync/` — body: {project_id: <pp_project_id>} — переносит записи ПП в задачи СП

### Задачи СП (Сводное планирование)
- `GET /api/tasks/?month=YYYY-MM&limit=200` — список задач за месяц
- `POST /api/tasks/` — создание задачи (work_name, work_num, department, sector, executor, date_start, date_end, deadline, plan_hours: {"2026-01": 40, ...}, project, stage, justification)
- `PUT /api/tasks/<id>/` — обновление задачи
- `DELETE /api/tasks/<id>/` — удаление задачи

### Зависимости задач
- `GET/POST /api/tasks/<id>/dependencies/` — список/создание зависимостей
  - POST body: {predecessor_id, dep_type: "FS"|"SS"|"FF"|"SF", lag_days: 0}
- `PUT/DELETE /api/dependencies/<id>/` — обновление/удаление
- `POST /api/tasks/<id>/align_dates/` — выравнивание дат {cascade: true|false}
- `GET /api/dependencies/?context=plan|pp&project_id=<id>` — все зависимости (для Ганта)

### Отчёты
- `GET /api/tasks/<task_id>/reports/` — список отчётов задачи
- `POST /api/tasks/<task_id>/reports/add/` — создание отчёта (doc_type, doc_number, doc_name, sheets_a4, date_start, date_end, norm, coeff, bvd_hours, ii_pi: "ИИ"|"ПИ", designation)
- `PUT /api/tasks/<task_id>/reports/<id>/edit/` — обновление
- `DELETE /api/tasks/<task_id>/reports/<id>/delete/` — удаление

### Журнал извещений
- `GET /api/journal/?per_page=100` — список извещений
- `POST /api/journal/` — ручное создание (notice_number, date_issued, date_expires, designation, work_name, ii_pi, department, sector, executor_name, closure_notice_number, closure_date_issued, closure_executor)
- `PUT/DELETE /api/journal/<id>/` — обновление/удаление

### Отпуска
- `GET /api/vacations/?month=YYYY-MM` — список отпусков
- `POST /api/vacations/` — создание (employee_name, date_from, date_to, vacation_type: "annual"|"sick"|"unpaid")

### Производственный календарь
- `GET /api/work_calendar/?year=2026` — нормы часов
- `POST /api/work_calendar/` — создание записи (year, month, hours_norm)
- `PUT/DELETE /api/work_calendar/<id>/` — обновление/удаление

### Справочники
- `GET /api/directories/?type=task_type` — список значений справочника
- `POST /api/directories/` — создание (type, value, parent_id)

### Настройки колонок
- `POST /api/col_settings/` — сохранение настроек (show_all_depts: true|false, ...)

## Архитектура симулятора

### Модуль 1: Генератор начальных данных (setup)
Создать реалистичную оргструктуру:
- 2 НТЦ-центра
- 4-6 отделов (по 2-3 на центр), коды вида "021", "028", "084", "086", "301"
- 2-3 сектора в каждом отделе
- 20-40 сотрудников с реалистичными русскими ФИО
- Роли: 1 admin, 2 ntc_head, 4-6 dept_head, 6-12 sector_head, остальные — user
- Производственный календарь на 2026 год (нормы рабочих часов по месяцам: Янв=120, Фев=152, Мар=168, Апр=176, Май=143, Июн=167, Июл=184, Авг=168, Сен=176, Окт=176, Ноя=159, Дек=183)

### Модуль 2: Планирование (начало года)
- Создать 3-5 проектов УП с кодами и названиями (напр. "РОС", "Персей")
- Для каждого проекта: 1-2 ПП-плана
- Для каждого ПП: 30-80 строк с реалистичными данными:
  - work_designation вида "АБВГ.ХХХХХХ.ХХХ"
  - Этапы 1-5, Вехи 1-8, Работы 1-20
  - Даты в пределах 2026 года
  - Типы: 60% "Выпуск нового документа", 20% "Корректировка документа", 15% "Техническое сопровождение", 5% другие
  - labor (трудоёмкость): 10-500 чел-часов
  - norm, coeff, sheets_a4, total_2d, total_3d — реалистичные значения
- Синхронизировать ПП→СП через API

### Модуль 3: Зависимости
- Создать 20-40% связей между задачами:
  - FS (Finish-to-Start): 70% связей
  - SS (Start-to-Start): 15%
  - FF (Finish-to-Finish): 10%
  - SF (Start-to-Finish): 5%
  - Lag 0-5 рабочих дней
- Убедиться что нет циклов
- Проверить конфликты дат через GET dependencies
- Для конфликтующих — вызвать align_dates

### Модуль 4: Месячная симуляция (январь-декабрь)
Для каждого месяца виртуального года:

1. **Планирование отпусков** (вероятность 5% на сотрудника/месяц):
   - Создать 1-2 недели отпуска
   - Проверить конфликт с задачами

2. **Создание новых задач** (5-15 задач/месяц):
   - СП-задачи создаются dept_head/sector_head
   - Назначение на сотрудников своего отдела/сектора
   - plan_hours распределяются по месяцам

3. **Обновление задач** (10-30 обновлений/месяц):
   - Изменение сроков, исполнителей, плановых часов
   - Проверка optimistic locking (отправка updated_at)

4. **Создание отчётов** (20-50% задач за месяц):
   - Для задач с task_type "Корректировка документа" — проверить автосоздание извещения
   - ii_pi: "ИИ" для новых, "ПИ" для замещающих
   - Проверить что doc_number не дублируется

5. **Работа с извещениями** (5-10/месяц):
   - Закрытие старых извещений (closure_notice_number, closure_date_issued)
   - Проверка статусов: active, expired, closed_yes, closed_no

6. **Корректировка ПП** (2-5 изменений/месяц):
   - Добавление новых строк
   - Обновление сроков существующих
   - Повторная синхронизация с СП

7. **Проверки целостности** (после каждого месяца):
   - Все задачи с зависимостями не имеют конфликтов (или конфликты зафиксированы)
   - Загрузка сотрудников не превышает нормы часов * 1.1 (допуск 10%)
   - Все извещения для "Корректировка документа" автосозданы
   - Отсутствие 500 ошибок в ответах API

### Модуль 5: Конкурентность
- Симулировать 3-5 одновременных пользователей (разные роли)
- Использовать asyncio + aiohttp для параллельных запросов
- Проверять:
  - Optimistic locking работает (409 при конфликте)
  - Rate limiting не блокирует нормальную работу
  - Visibility filter: user видит только свои задачи, dept_head — задачи отдела

### Модуль 6: Отчётность и метрики
По завершении симуляции вывести:
- Общее количество созданных: проектов, ПП, задач, отчётов, извещений, отпусков, зависимостей
- Ошибки API (коды 4xx, 5xx) с группировкой по endpoint
- Среднее время ответа API по endpoint
- Найденные баги (несоответствие ожидаемого и фактического поведения)
- Проверки бизнес-логики:
  - Количество конфликтов зависимостей
  - Перегруженные сотрудники (часы > нормы)
  - Извещения без автосоздания
  - Задачи с просроченными сроками

## Технические требования

1. **Python 3.11+**, библиотеки: `requests`, `asyncio`, `aiohttp`, `faker` (для русских ФИО)
2. **Ускорение**: 1 виртуальный день = 2-3 секунды реального времени (365 дней ≈ 12-18 минут на основной цикл + время на setup и анализ)
3. **Логирование**: файл `simulation.log` + прогресс в stdout (текущий виртуальный месяц, количество операций)
4. **Конфигурация**: файл `sim_config.yaml` с параметрами (URL сервера, количество сотрудников, интенсивность операций)
5. **Воспроизводимость**: фиксированный random seed для повторяемых результатов
6. **Отказоустойчивость**: программа продолжает работу при ошибках API (логирует и идёт дальше)
7. **Итоговый отчёт**: `simulation_report.json` с полными метриками

## Особенности бизнес-логики

1. Работа может быть одновременно в ПП и СП (`show_in_pp=True, show_in_plan=True`)
2. При синхронизации ПП→СП создаётся связь через `actions.pp_id`
3. Заблокированные поля PP-задач в СП: work_name, work_num, work_designation, stage, justification
4. Извещение автоматически создаётся при отчёте с task_type="Корректировка документа"
5. Зависимости: FS = начало после окончания, SS = начало после начала, FF = окончание после окончания, SF = окончание после начала
6. Лаг в зависимостях считается в рабочих днях (пн-пт)
7. Производственный календарь: hours_norm задаёт норму часов на месяц
8. Роль `is_writer` = все кроме `user`; `user` может только читать
9. ntc_head/ntc_deputy с `show_all_depts=True` видят задачи всех отделов
10. Удаление задачи из СП, если она из ПП — только убирает `show_in_plan=False`

```
ntc_simulator/
├── sim_config.yaml          # Конфигурация

├── api_client.py            # HTTP-клиент с сессиями

│   └── business_rules.py    # Проверки бизнес-правил
├── report_generator.py      # Генерация итогового отчёта
└── simulation.log           # Лог выполнения
```

## Пример запуска

```bash
# Убедиться что сервер запущен
cd planapp_django && python manage.py runserver 8001

# В другом терминале
cd ntc_simulator
pip install requests aiohttp faker pyyaml
python main.py --seed 42 --speed 3  # speed: секунд на виртуальный день
```

## Критерии успеха

1. Симуляция проходит полный год без критических ошибок (0 ошибок 500)
2. Все бизнес-правила соблюдаются (автосоздание извещений, блокировка PP-полей, видимость по ролям)
3. Конкурентные операции корректно обрабатываются (optimistic locking, rate limiting)
4. Итоговый отчёт содержит метрики для оценки стабильности системы
5. Время выполнения не превышает 1 час
